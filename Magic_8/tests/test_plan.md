# Тест-план проекта "Магический шар"

## 1. Введение
**Описание проекта:**  
Проект реализует "Магический шар", который отвечает на любые вопросы пользователя случайным образом из заранее подготовленного списка. Программа поддерживает:  
- Ввод имени пользователя и приветствие  
- Задание одного или нескольких вопросов  
- Ответы на вопросы из списка заранее подготовленных вариантов  
- Завершение работы по желанию пользователя  

**Цель тестирования:**  
Обеспечить корректную работу программы, проверку правильности выдаваемых ответов, обработку некорректного или пустого ввода и корректное завершение работы.  

**Особенности тестирования:**  
- Фокус на функциональности генерации случайного ответа  
- Проверка корректного взаимодействия с пользователем через консоль  
- Негативные тесты на пользовательский ввод  

---

## 2. Область тестирования
**Покрываемые модули:**  
- Модуль ввода имени пользователя  
- Модуль ввода вопросов  
- Модуль генерации случайного ответа (`random_pick`)  
- Логика продолжения или завершения работы программы  

**Исключаем:**  
- Работа с внешними файлами  
- Сетевые операции и API (проект локальный)  
- UI-тестирование (консольная программа)  

---

## 3. Тестовая стратегия
**Типы тестирования:**  
- **Функциональное:** проверка корректного вывода ответов на вопросы  
- **Негативное тестирование:** проверка реакции на пустой ввод имени или вопроса, неправильный ввод при подтверждении продолжения  
- **Граничные значения:** пустое имя пользователя, пустой вопрос, ввод символов вместо букв  
- **Исследовательское тестирование:** тестирование различных комбинаций вопросов и ответов  

**Приоритет тестов:**  
1. Критические: корректная выдача ответа, падение программы при пустом вводе  
2. Средний: обработка нестандартных символов в вопросах  
3. Низкий: очень длинный вопрос  

---

**Staging:**  
- Консольная версия программы на отдельной ветке  
- Тестирование стабильных сборок перед публикацией  

**Prod:**  
- Финальная версия проекта для демонстрации  
- Не требует внешних подключений

## 5. Подход к автоматизации
На данном этапе автоматизация не проводится.

## 6. Баг-репорты
**Оформление багов:**  
- Уникальный ID (BUG-001, BUG-002…)  
- Название 
- Подробное описание
- Требование 
- Ожидаемый результат  
- Фактический результат  
- Шаги воспроизведения  
- Окружение
- Severity
- Priority
- Вложения
- Статус
- Связанные артефакты 

**Хранение:**  
- Markdown-файл `bug_reports.md` в репозитории  

**Приоритизация:**  
- Высокий: падение программы, критические ошибки логики  
- Средний: некорректная обработка ввода пользователя  
- Низкий: редкие или малозначимые баги  

---

## 7. Риски и ограничения
**Риски:**  
- Падение программы из-за некорректного пользовательского ввода 
- Ошибки генерации случайного ответа  
- Пустой вывод при пустом вопросе или имени  

**Ограничения:**  
- Нет UI/UX тестирования, т.к. проект консольный
- Нет API или базы данных
