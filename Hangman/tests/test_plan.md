# Тест-план проекта "Виселица"

## 1. Введение
**Описание проекта:**  
Проект реализует игру "Виселица" на Python с поддержкой русского языка. Пользователь угадывает слово, вводя буквы или полное слово. Программа отслеживает прогресс и отображает виселицу.  

**Поддерживаемые функции:**  
- Угадывание букв и слов  
- Сохранение введенных букв и слов  
- Проверка повторного ввода  
- Отображение прогресса слова и виселицы  
- Обработка некорректного ввода (цифры, спецсимволы, пустая строка)  

**Цель тестирования:**  
Обеспечить корректный игровой процесс, проверку ввода пользователя и правильное завершение игры (победа/проигрыш).  

**Особенности тестирования:**  
- Фокус на функциональности игрового процесса  
- Проверка обработки букв, слов, регистра и спецсимволов  
- Негативные тесты на пользовательский ввод  

---

## 2. Область тестирования
**Покрываемые модули:**  
- Функция `play(word)` (основная логика игры)  
- Функция `display_hangman(tries)` (визуализация виселицы)  
- Функция `get_word()` (случайный выбор слова)  

**Исключаем:**  
- Работа с внешними файлами  
- Сетевые операции и API (проект локальный)  
- Графический интерфейс (консольная программа)  

---

## 3. Тестовая стратегия
**Типы тестирования:**  
- **Функциональное:** проверка корректного угадывания букв и слов, правильное отображение прогресса и виселицы  
- **Негативное:** проверка реакции на цифры, спецсимволы, пустой ввод, повторный ввод  
- **Граничные значения:**  
  - 0 попыток (поражение)  
  - Последняя попытка (6-я ошибка)  
  - Слова с повторяющимися буквами  
- **Исследовательское тестирование:** выявление неожиданных сценариев, например смешанный ввод букв и слов  

**Приоритет тестов:**  
1. Критические: корректное угадывание букв и слов, завершение игры, падение при пустом вводе  
2. Средний: обработка спецсимволов, регистра букв  
3. Низкий: редкие сценарии (очень длинные слова)

## 6. Баг-репорты
**Оформление багов:**  
- Уникальный ID (BUG-001, BUG-002…)  
- Название  
- Окружение
- Связанные артефакты 
- Шаги воспроизведения  
- Ожидаемый результат  
- Фактический результат  
- Severity
- Priority
- Обоснование

**Хранение:**  
- Markdown-файл `bug_reports.md` в репозитории  

**Приоритизация:**  
- Высокий: падение программы, критические ошибки функционала  
- Средний: обработка повторного ввода, регистра букв, спецсимволов  
- Низкий: визуальные или редкие баги

## 7. Риски и ограничения
**Риски:**  
- Падение программы из-за некорректного пользовательского ввода
- Повторный ввод уже угаданных букв или слов  
- Потеря регистра или некорректное отображение виселицы  

**Ограничения:**  
- Нет UI/UX тестирования, т.к. проект консольный
- Нет API или базы данных
